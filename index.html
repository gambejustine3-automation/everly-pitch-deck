The Problem

Manual workflows
kill creative businesses

Photography studios lose clients not because of poor photography — but because of slow responses, missed follow-ups, and inconsistent operations. Every manual task is a failure point waiting to happen.

This system was built to eliminate that entirely — replacing every manual touchpoint with a reliable, consistent, AI-powered process that works even while the photographer is on a shoot.

01 / 04

Lost Leads

Without instant AI triage, HOT leads go cold within hours. Studios manually sifting through inquiries lose clients to faster competitors who respond in minutes.

02 / 04

Inconsistent Communication

Every email written from scratch means different quality, tone drift, and forgotten follow-ups. A no-show discovery call might never get a rebook email. Revenue disappears silently.

03 / 04

Invisible Pipeline

Without real-time pipeline tracking, no one knows where each client actually is — leading to double bookings, missed invoices, and delayed deliveries.

04 / 04

Post-Delivery Revenue Leakage

Most studios deliver the gallery and stop. No review requests, no rebooking sequence, no LTV tracking. Repeat business — the most profitable revenue stream — never materializes.

Technical Depth

Engineering decisions
worth noting

Five examples of deliberate technical thinking behind the system — not just "connecting tools", but building reliably at every layer.

Structured AI Outputs

Schema-Enforced GPT Responses

Every AI step uses a defined outputSchema — not free-form text parsing. This means Zapier can map AI outputs like lead_status and urgency_score as discrete fields in downstream steps — eliminating parsing errors entirely.

// Output schema enforced on every AI step outputSchema: { "lead_status": "", "urgency_score": "", "summary": "", "recommended_action": "" }
ID Generation System

Atomic Counter with Config Sheet

Lead IDs and Project IDs use a Config sheet counter to prevent duplicates. Zapier reads the current counter, increments by 1, and writes back before creating the record — a lightweight atomic operation without a database.

# System 2 — Project ID Generator from datetime import datetime year = datetime.now().year suffix = lead_id.replace("LED-", "")[-4:] project_id = f"EVR-{year}-{suffix}" # → EVR-2026-0042
Pricing Engine

4-Tier Fallback Price Resolution

System 3A's Python pricing calculator uses a strict priority chain: (1) operator-confirmed amount via /setbudget for $10,000+/TBD leads, (2) client-selected budget from the form ($1k–$10k fixed options), (3) PRICE_MAP by package name, (4) raw budget string parsing. Client budget always takes precedence over package defaults.

BUDGET_MAP = { "$1,000": 1000, "$3,000": 3000, "$5,000": 5000, "$7,500": 7500, "$10,000": 10000, } deposit = round(total * 0.30) balance = total - deposit
Webhook Architecture

Backend Bot on Railway.app

A custom Python server on Railway.app serves as the command router. Victoria sends pipeline commands from Telegram that fire Zapier webhooks — decoupling the CRM layer from the automation layer entirely.

# Railway endpoints across systems POST /pipeline_notify # Sys 1.5A POST /proposal_notify # Sys 2 POST /invoice_sent # Sys 3B POST /gallery_notify # Sys 4 POST /retention_notify # Sys 5A POST /retention_5b_notify # Sys 5B
Production Reliability

Idempotency Guards + Budget Gate

Deposit and balance payment buttons include idempotency checks — if already marked paid, re-pressing does nothing and notifies the operator. The budget gate blocks System 3A from firing on uncertain budgets ($10,000+/TBD) until the operator confirms the exact amount via /setbudget.

# Idempotency guard — prevent double-marking if deposit_paid_flag.upper() == "TRUE": answer_callback(cb_id, "Already marked!") return # silent no-op # Budget gate for uncertain amounts UNCERTAIN_BUDGETS = {"$10,000+", "TBD", "10,000+"} if budget in UNCERTAIN_BUDGETS: prompt_setbudget(lead_id) # block 3A
